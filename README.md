# TMService
SNTP win32 client/server with smart time control.
Using system timer speed change for soft time correction.

File `IdSNTPX.pas` distributed under original licenses from http://www.indyproject.org/

---

## Описание

TMService — фоновое Windows-приложение (x32), которое поддерживает точность системных часов, управляя **скоростью хода** аппаратного таймера через `SetSystemTimeAdjustment`, а не просто переставляя время скачком. Дополнительно ведёт виртуальный высокоточный PFC-таймер (на основе TSC) для независимого отсчёта времени.

Архитектура:
- `TProfileTimer` (PFC-таймер) — высокоточный виртуальный таймер на счётчике TSC.
- `SetSystemTimeAdjustment` — управление скоростью системных часов Windows.
- NTP-клиент (Indy `TIdSNTP`) — синхронизация с пулом NTP-серверов.
- Встроенный SNTP-сервер — раздача времени другим хостам в сети.

#### Режимы коррекции часов

TMService работает в одном из двух режимов на каждом цикле проверки. Режим выбирается сравнением текущего смещения PFC↔системные часы (`|dta_ms|`) с `min_dvg` и `max_dvg` из секции `[bounds]`.

---

**Режим плавной подстройки скорости** — активен когда `|dta_ms| < min_dvg` (по умолчанию 125 мс)

Смещение достаточно мало, чтобы корректировать его постепенно без скачка времени. Алгоритм вычисляет сигнал коррекции скорости `ref_dta` (мс/ч) и применяет его к `st_adjust` — аргументу `SetSystemTimeAdjustment`. Это заставляет таймер Windows тикать чуть быстрее или медленнее до тех пор, пока смещение не поглотится.

Параметры, управляющие агрессивностью коррекции в этом режиме:

| Ситуация | Параметр | Эффект |
|---|---|---|
| NTP **не** опрашивался в этом цикле | `PFCTrust` | Масштабирует `ref_dta` на 0–1. Ограничивает избыточную коррекцию при шумном якоре PFC. |
| NTP **опрашивался** в этом цикле | `NTPAproxLow/High`, `NTPNuclearMs`, `NTPStreakCount` | Кривая доверия NTP — масштабирует `ref_dta` в зависимости от `\|ntp_ms\|`. См. раздел ниже. |
| Зона точной подстройки `\|dta_ms\| < 1 мс` | `RefDtaEmaTC` | Сглаживает `ref_dta` через быстрый EMA для подавления шума измерения. |
| EMA базовой скорости для `st_adjust` | `EmaSpeedMode`, `EmaSpeedThr` | Фиксирует стабильную рабочую точку `st_adjust` для «примагничивания» при переходе через ноль. |

Подстройка скорости **заблокирована** при срабатывании жёсткой коррекции (режимы взаимоисключающие).

---

**Режим жёсткой коррекции** — активен когда `min_dvg ≤ |dta_ms| ≤ max_dvg`

Смещение слишком велико для плавной подстройки скорости. Вызывается `IndySetLocalTime` — системные часы скачком переводятся к значению PFC-таймера. `st_adjust` при этом **не меняется** — корректируется только позиция. После вызова история дрейфа (`clk_pfc_dev`) сбрасывается, чтобы регулятор скорости перезапустился чисто.

Параметры, управляющие жёсткой коррекцией:

| Параметр | Эффект |
|---|---|
| `AproxFactor` | Доля смещения, применяемая за один цикл: `dta := dta × AproxFactor`. При 0.87 за одну минуту корректируется 87% смещения; остаток сходится в последующих циклах. Предотвращает перерегулирование. |
| `min_dvg` | Минимальное смещение для запуска жёсткой коррекции. Ниже этого значения работает только подстройка скорости. Меньшие значения расширяют зону мягкого режима. |
| `max_dvg` | Максимальное смещение, выше которого жёсткая коррекция тоже подавляется (защита от аномалий). |

Если смещение стабильно превышает `min_dvg` (например из-за скачка TSC или события питания), жёсткая коррекция срабатывает каждый цикл до тех пор, пока позиционная ошибка не поглотится, после чего система возвращается в режим плавной подстройки скорости.

#### PFCTrust — коэффициент доверия якорю (мягкий режим)

PFC-таймер (TSC-based) используется как **якорь**: именно его показания определяют, насколько системные часы спешат или отстают (`dta_ms`), и на их основе вычисляется коррекция скорости (`ref_dta → st_adjust`). Пока якорь стабилен — это работает хорошо. Но якорь бывает нестабилен:

- **Виртуальные машины** (Hyper-V, VMware, VirtualBox) — гипервизор может приостанавливать TSC при миграции vCPU между ядрами или при паузе VM. Скачки TSC выглядят как «дрифт» системных часов и вызывают ложные коррекции.
- **Троттлинг CPU** (SpeedStep, C-states) — при изменении частоты процессора счётчик TSC на части платформ некорректно масштабируется, добавляя шум ±1–5 мс/мин.
- **Многосокетные системы** — TSC разных сокетов может расходиться.

**Проблема:** если PFC-таймер сам «плывёт» на ±3 мс/мин, то измеренный `dta_ms` содержит этот шум. Алгоритм коррекции принимает его за реальный уход системных часов и начинает менять `st_adjust` в ответ на шум якоря — в результате `st_adjust` осциллирует, и реальная точность системного времени ухудшается.

**Пример с нестабильным PFC:**

```
Реальный уход системных часов:      +2 мс/ч
Шум PFC-таймера:                    ±180 мс/ч  (±3 мс/мин)
Измеренный dta_ms (система vs PFC): ±182 мс/ч  — в основном шум
Без PFCTrust: алгоритм гонится за шумом, st_adjust ±5000 единиц
С PFCTrust=0.1: коррекция ×0.1, st_adjust ±500 — менее агрессивно
```

**Значения параметра:**

| `PFCTrust` | Поведение |
|---|---|
| `1.0` | Полное доверие. Коррекция применяется в полном объёме. Оптимально когда PFC стабилен (физическая машина, modern Intel/AMD). |
| `0.5` | Половина коррекции без NTP. Хороший компромисс для умеренно нестабильного PFC (лёгкий троттлинг, некоторые VM). |
| `0.1`–`0.2` | Минимальная коррекция без NTP. Для сильно нестабильного PFC (старый Hyper-V, VirtualBox). |
| `0.0` | Коррекция скорости часов заморожена до следующего NTP-синха. Только NTP-события двигают `st_adjust`. |

Параметр действует **только в режиме плавной подстройки** и **только когда в текущем цикле не было опроса NTP** (`ntp_sync = false`). После NTP-синхронизации масштабирование коррекции берёт на себя кривая доверия NTP (см. ниже). Оба механизма взаимоисключающие в рамках одного цикла и не действуют в режиме жёсткой коррекции.

---

#### Кривая доверия NTP — коррекция скорости при NTP-синхронизации (мягкий режим)

Когда `ntp_sync = true`, сигнал коррекции скорости (`ref_dta`) масштабируется коэффициентом, определяемым **величиной NTP-смещения** (`|ntp_ms|`). Цель — предотвратить избыточную коррекцию во время больших осцилляций и погоню за джиттером NTP когда часы почти точны.

**Четыре зоны, привязанные к `min_dvg`:**

```
|ntp_ms| < NTPAproxLowMs (3 мс)
    → коэффициент = NTPAproxLow (0.25)
    Зона скептицизма: вероятно доминирует джиттер NTP. Четверть веса.

NTPAproxLowMs ≤ |ntp_ms| < NTPNuclearMs (15 мс)
    → коэффициент = линейно NTPAproxLow → NTPAproxHigh (0.25 → 0.9)
    Нормальная зона: смещение реально. Коррекция масштабируется с доверием.

NTPNuclearMs ≤ |ntp_ms| < min_dvg (125 мс)  — полу-ядерная зона
    → коэффициент = NTPNuclearFloor (0.15)  [или 0.5 при серии ≥ NTPStreakCount]
    Жёсткая коррекция НЕДОСТУПНА (смещение ниже min_dvg).
    Небольшое подталкивание скорости удерживает систему от полной остановки.
    Мягкий коэффициент предотвращает осцилляцию, позволяя постепенно нагонять.

|ntp_ms| ≥ min_dvg (125 мс)  — полная ядерная зона
    → коэффициент = 0.0  [или 0.5 при серии ≥ NTPStreakCount]
    Жёсткая коррекция (IndySetLocalTime) берёт на себя позицию.
    Подстройка скорости поверх вызвала бы двойную коррекцию и осцилляцию.
    Только систематический дрейф (подтверждённая серия) возвращает мягкую подстройку.
```

Граница между полу-ядерной и полной ядерной зоной — это `min_dvg` из `[bounds]`, не отдельный параметр. Это гарантирует, что два механизма коррекции — подстройка скорости и жёсткий сброс позиции — всегда чётко разделены без мёртвой зоны между ними.

**Связь с `PFCTrust`:**
- `PFCTrust` действует когда **NTP-синхронизации не было** в этом цикле (неопределённость якоря).
- Кривая доверия NTP действует когда **NTP-синхронизация была** (неопределённость величины измерения).
- Два механизма взаимоисключающие в рамках одного цикла.

#### Защита от шума NTP — минимальный порог погрешности

Перед применением любой коррекции TMService проверяет, отличимо ли измеренное NTP-смещение от шума самого измерения. Если нет — шаг синхронизации пропускается полностью.

**Оценка пола измерения:**

```
meas_floor_ms = rtt_half_ms  +  pool_disp_ms
```

- `rtt_half_ms = min_RTD / 2` — половина минимального round-trip delay среди выбранных NTP-серверов. Это лучшая оценка одностороннего расхождения: даже на идеально симметричном пути смещение нельзя знать точнее чем RTD/2.
- `pool_disp_ms` — максимальное отклонение любого из выбранных серверов от средневзвешенного смещения. Отражает разногласие источников.

Сумма — это **пол измерения**: минимальная достижимая неопределённость смещения для данного цикла опроса. Любое измеренное смещение ниже неё неотличимо от шума.

**Правило принятия решения:**

```
|ntp_diver_ms| < meas_floor_ms  →  синхронизация пропущена (смещение внутри шума)
|ntp_diver_ms| ≥ meas_floor_ms  →  синхронизация выполняется штатно
```

При пропуске `ntp_diver` приравнивается к нулю, `ntp_sync` остаётся `false` на этот цикл. Кривая доверия NTP и `AdjustPFCTimerSpeed` не вызываются. Часы продолжают идти от PFC-таймера с масштабированием по `PFCTrust`.

**Практический эффект:** на хорошо настроенной системе с `|dta_ms| < 1 мс`, RTD пула NTP обычно 5–30 мс, что даёт `meas_floor ≈ 2.5–15 мс`. Смещение 0.5 мс будет ниже пола шума и не вызовет коррекции — стабильные часы не тревожатся без необходимости.

Все три компонента — `pool_disp`, `rtt_half` и `meas_floor` — логируются при каждом NTP-опросе на уровне `#DBG`.

### Секция `[bounds]`

Пороги, определяющие режим работы алгоритма коррекции.

| Параметр | Тип | По умолч. | Описание |
|---|---|---|---|
| `min_dvg` | float (мс) | `~16` (точность таймера) | Минимальное **абсолютное** смещение системных часов (в мс) при котором выполняется жёсткая коррекция времени (`IndySetLocalTime`). Ниже этого порога — только аппаратная подстройка скорости. |
| `max_dvg` | float (мс) | `1800000` (30 мин) | Максимальное допустимое смещение для жёсткой коррекции. Если смещение превышает это значение — коррекция считается аномальной и НЕ выполняется (требует ручного вмешательства). |
| `max_rqt` | float (мс) | `1000` | Максимальное время NTP round-trip (мс). Ответы с RQT выше этого значения отбрасываются как ненадёжные. |
| `min_date` | дата | сегодня | Минимальная допустимая дата системных часов. Если дата системы меньше — считается аномалией. |

#### Взаимодействие `min_dvg` / `max_dvg` с алгоритмом

```
|dta_ms| < min_dvg  → sync_expected = FALSE → hw_adjust (подстройка скорости),
                                               жёсткой коррекции нет
|dta_ms| in [min_dvg, max_dvg] → sync_expected = TRUE → жёсткая коррекция времени
                                                         hw_adjust НЕ выполняется (*)
|dta_ms| > max_dvg  → sync_expected = FALSE → hw_adjust, жёсткой коррекции нет
```

(*) — известное ограничение: при постоянном дрейфе > min_dvg аппаратная коррекция скорости
блокируется, и программа выполняет жёсткий сброс времени каждую минуту вместо плавной
подстройки. Уменьшение `min_dvg` (например до 50 мс) расширяет зону работы hw_adjust.

## Консольные сообщения

TMService использует префиксы для обозначения уровня сообщений. Понимание их помогает отличить нормальную работу от реальных проблем.

### Легенда префиксов

`#DBG` — отладка/трассировка, подробное внутреннее состояние. Норма; можно подавить снижением `log_verbose_level` (клавиша **V**).
`#MSG` — информационное. Нормальные рабочие события (результат NTP-запроса, выполнена синхронизация и т.д.).
`#LOADCFG` — конфигурация загружена или перезагружена. Появляется при старте и при нажатии **C**.
`#RELOAD` — перезагрузка конфига инициирована клавишей **C**.
`#STAT` — статистика peer-синхронизации (только при `EnableSyncStat=1`).
`#WARN` — состояние, требующее внимания, но сервис продолжает работу. См. таблицу ниже.
`#ERR` / `PrintError` — восстанавливаемая ошибка.

### Частые сообщения `#WARN`

**`#WARN: g_timer not sync, hurry diff = X ms`**
Виртуальный таймер (`TVirtualTimer`, разделяемая память `Global\VIRTUALTIMER`) отклонился более чем на 1 мс от опорного PFC-якоря. Сервис автоматически корректирует скорость таймера (`DiffTimer.PFCRatio`) в том же шаге. Периодическое появление нормально в первые несколько циклов VTT-синхронизации после запуска или после жёсткой коррекции времени. Постоянное появление (каждый цикл синхронизации) означает, что период VTT слишком короткий или сам PFC-таймер нестабилен — проверьте `VTSyncRate` и `PFCTrust`.

**`#WARN: g_timer.OwnRights = false`**
Сервис не имеет прав владельца на блок разделяемой памяти таймера. В нормальной работе сервиса не должно появляться — означает что другой процесс владеет блоком или сервис работает в режиме читателя. Проверьте, не запущен ли второй экземпляр.

**`#WARN: not have rights for g_timer`**
Аналогично предыдущему, но логируется из пути подстройки часов. Обновление разделяемого блока было пропущено.

**`#WARN: NTPServer feature switched off — may be UDP port 123 already used`**
Привязка встроенного SNTP-сервера к UDP-порту 123 не удалась (скорее всего уже запущен другой NTP-демон). Клиентская синхронизация NTP не затронута. Либо остановите конфликтующий сервис, либо установите `EnableNTPServer=0`.

**`#WARN: Hard time syncrhonizing — more than one week delta correction`**
NTP вернул время более чем на 7 дней от текущих системных часов. Сервис выполняет жёсткий `SetLocalTime` к значению NTP. Ожидается после длительной работы без сети или при сбросе RTC из-за разряда батарейки.

**`#WARN: Time response HH:MM:SS overrides with HH:MM:SS, due 5 seconds error`**
Ранее принятый NTP-ответ вытеснен более поздним, более согласованным ответом, отличающимся более чем на 5 секунд. Используется более позднее значение. Может появляться когда первый опрошенный сервер сильно рассинхронизирован, а второй возвращает корректное время.

**`#WARN: Cannot open timer section, trying to create new`** *(DateTimeTools)*
Слот разделяемой памяти для VirtualTimer не удалось открыть; создаётся новый слот. Нормально при самом первом запуске или при несовпадении индекса слота. Ненормально если повторяется при каждом запуске — проверьте права на именованные объекты `Global\`.

### Клавиши управления (окно консоли)

| Клавиша | Действие |
|---|---|
| **E** | Плавное завершение работы |
| **V** | Переключение уровня подробности лога (0 → 1 → … → 7 → 0) |
| **C** | Перезагрузка конфигурации из INI-файла (включая список NTP-серверов) |
| **+** / **−** | Увеличить / уменьшить `st_adjust` на 15 мс/ч и немедленно применить |

## Исправленные баги (история)

### v1.0.2.28+ — нестабильность st_adjust при устойчивом дрейфе

**Симптом:** при дрейфе системных часов >125 мс (> min_dvg) программа каждую минуту
выполняла жёсткий сброс времени вместо плавной подстройки скорости. Через 20–30 минут
начиналась осцилляция ±300 мс с периодом 2 минуты.

**Причина A** (`prv_dev := 0` при `sync_expected`): каждая жёсткая синхронизация сбрасывала
историю предыдущей скорости дрейфа. Следующий вызов `hw_adjust` видел `prv_dev = 0` и
удваивал коррекцию с нуля → `st_adjust` рос неограниченно → overshooting.

**Причина B** (отсутствие лимита шага): при коротком `last_chk_elps` или после `prv_dev = 0`
вычисленный `ref_dta` мог превышать 10 000 мс/ч за одну итерацию, что гарантировало
overshooting.

**Исправление A:** убран сброс `prv_dev := 0` — значение сохраняется между синхронизациями.

**Исправление B:** добавлено ограничение разового шага: `if Abs(ref_dta) > 3000 then ref_dta := Sign(ref_dta) * 3000`.
